## Context

`solo-tetris-mvp`（已归档）交付了完整的单机地基，并为联机预留了关键边界：
- `shared/` 层全部纯逻辑、零 DOM/网络依赖，可被 server 直接 import（有 purity 测试保证）
- 可种子化 7-bag（`mulberry32` + `createBagQueue`）——相同种子产生相同方块序列
- `engine` 为 reducer 风格（`createGame/step/dispatch`），可手动驱动、可测、可重放
- 能量积累已实现，但 spec 明确"本阶段不消耗"——联机需解开这道封印
- `client/main.ts` 一进来就直接进单机对局，`client/ui/` 空置，`server/` 仅占位

本 change 在此之上叠加房间 + 实时多人同步 + 能量攻击。

## Goals / Non-Goals

**Goals:**
- 2–4 人同屏联机对战，**自己棋盘本地跑**（保留单机的低延迟手感）
- 能量攻击互动（释放垃圾行给对手），最后存活者获胜
- 最大化复用 `shared/`，不重写任何游戏规则
- 对手画面"大体"投影即可，无需竞技级同步

**Non-Goals:**
- 服务器权威 / 反作弊加固（信任客户端，朋友间自用）
- AI 接管掉线者、观战、聊天、排行榜、账号
- 对手活动方块的实时投影（只投影已堆叠状态）
- 跨网络时钟的纳秒级同步

## Decisions

### D1. 混合权威：自己棋盘本地权威 + 共享种子 + 攻击走服务器
每个客户端本地跑自己的 `engine`；开局服务器下发同一个 `seed`，所有人 `createGame(seed)`（公平）；只有"攻击"这种跨玩家交互经服务器仲裁。
- **为什么**：俄罗斯方块对输入延迟零容忍，服务器权威（输入送服务器算）会让手感卡顿，不可接受；纯 P2P 有 NAT/可靠性难题。
- **替代**：服务器权威 → 延迟高，否决。纯 P2P → 连通性/抗丢包复杂，否决。

### D2. Socket.io + rooms
服务器用 Node + Socket.io，房间用其原生 `rooms` 能力（join/leave/广播到房间）。
- **为什么**：房间管理、断线重连、房间广播都是现成的，省大量轮子。
- **替代**：原生 `ws` → 房间/重连/心跳全要自写，否决。

### D3. 房间 ID：6 位纯数字，服务器查重
`000000`–`999999`，生成时查重避免冲突。
- **为什么**：纯数字最易口耳相传/输入（对比 UUID 难念、字母数字易混 0/O）。
- **替代**：UUID（难输入）、字母数字（易混淆），否决。

### D4. 对手投影 = 锁定时事件驱动广播压缩快照（不含活动方块）
玩家每次**锁定方块**（引擎 `lockNow` 那一刻）广播自己棋盘的压缩快照；接收方更新对手预览（只画已堆叠格子）。
- **为什么**：锁定是棋盘"已堆叠状态"唯一变化的时刻，且频率=玩家放块节奏（约 0.5–2s/次），带宽极低；"大体即可"不需要插值/预测。
- **替代**：定时广播（无变化也发，浪费）；操作流重放（需严格确定性+丢包重传，复杂）；额外低频广播活动方块（多流量，非必需）。均否决。

### D5. 快照压缩格式
10×20 棋盘，每格 8 状态（7 方块 + 空）= 3 bit。简单实现用每格 1 字节（200 字节/快照）或 nibble 打包（~100 字节）。MVP 可先用字符串/字节数组，带宽充裕后再优化。
- **为什么**：锁定频率低（≤2 次/秒 × N 对手），即使 200 字节/次也远低于任何带宽门槛。

### D6. 能量消耗：`spendEnergy` 纯函数，消耗 5 = 1 单位，信任客户端
按 U 时若能量 ≥5 则消耗 5、释放 1 次攻击；<5 则失败。消耗逻辑放 `shared/energy.ts`（纯函数）。客户端自报能量状态，服务器**信任**但**限频**（防刷）。
- **为什么**：朋友间自用，信任客户端开发最快；限频足以挡住最粗暴的刷攻击。
- **替代**：服务器用 `shared/energyGain` 按客户端上报的消行事件重算能量并校验 → 严谨但复杂，本 change 不做（留后续）。

### D7. 垃圾行：服务器随机缺口列 + 受害者 buffer queue + 下次锁定顶入
- **缺口列**由**服务器随机**决定（不让攻击者定向卡死某列，标准做法）
- 垃圾行不立即顶入，进受害者的 **buffer queue**；受害者**下次锁定**时把队列里的垃圾行顶入棋盘底部
- **为什么 buffer**：给受害者一个反击窗口（"我再消一行就把攻击打回去"），避免被瞬间顶死的挫败感；这是现代对战俄罗斯方块的核心手感。
- 垃圾行逻辑放新增 `shared/garbage.ts`（纯函数：缺口随机、buffer 操作、顶入 merge），client/server 共用。
- **替代**：立即顶入（挫败感强）、定时顶入（时序复杂），否决。

### D8. 目标选择：QE 手动循环 + 智能默认
Q/E 在存活对手间循环切换目标；默认指向"上一个攻击自己的人"，无人攻击过则指向第一个存活对手。
- **为什么**：手动选择有策略和互动（符合原始需求），智能默认让不切目标也有意义（复仇）。
- **替代**：纯随机（不可控、不爽）、自动瞄准最危险者（少策略），否决。

### D9. 掉线 = 淘汰，最后存活者获胜
socket 断开即视为该玩家淘汰；剩余玩家继续；只剩一人时该人获胜。
- **为什么**：最简单可靠。
- **替代**：AI 接管（复杂）、冻结其棋盘（复杂），否决。

### D10. 开局同步：房主开始 → 服务器生成 seed → 倒计时 → 同步开局
房主点开始 → 服务器生成 `seed` + 玩家列表，广播 `gameStart{seed, players}` → 客户端显示 3-2-1 倒计时（由服务器 start 时间戳驱动，避免时钟漂移）→ 倒计时结束各端 `createGame(seed)` 同时开始。

### D11. 人数可变 2–4，UI 优化 3 人
创建房间时选 2/3/4 人；对手预览按存活对手数组自适应渲染（3 人左右对称最美，4 人右侧堆叠）；代码不硬编码人数。

### D12. shared 扩展保持向后兼容
`energy` 加 `spendEnergy`（纯函数）；新增 `garbage.ts`（纯函数）。均为纯增量，单机路径不调用，单机测试不受影响。

## Risks / Trade-offs

- **[网络抖动致对手预览延迟]** → 可接受（"大体即可"）；锁定事件最终会到达，预览最多滞后一两个方块。
- **[短暂断线被误判淘汰]** → MVP 直接淘汰；若体验差，后续加 3s 重连宽限期。
- **[客户端伪造能量刷攻击]** → 信任客户端的已知风险；服务器对每玩家攻击**限频**（如 ≤2 次/秒）缓解。
- **[房间 ID 碰撞]** → 服务器生成时查重。
- **[开局时钟不同步]** → 倒计时由服务器 start 时间戳驱动，各端据此计算本地开始时刻。
- **[单机回归]** → shared 扩展纯增量；client 单机路径不触碰网络层；现有 107 测试须持续通过。
- **[服务器单点]** → 自用规模单进程足够；不做水平扩展。

## Migration Plan

- **server/**：新建 Node + Socket.io 服务，独立进程/端口部署；不影响前端构建。
- **client/**：加菜单状态机，单机路径保持不变（菜单→单机 即原 main.ts 逻辑）。
- **shared/**：纯增量扩展（`spendEnergy`、`garbage.ts`），单机测试全绿即视为兼容。
- 无数据迁移、无线上流量（全新功能）。

## Open Questions

- **掉线宽限期**：MVP 倾向不要（直接淘汰），若手玩体验差再加 3s 重连窗口。
- **房间最低开局人数**：建议 2 人即可开始（房主可单独练习等待，或 2 人即开打）。
- **攻击限频具体值**：建议每玩家 ≤2 次/秒（能量自然限制下基本不会触及，纯防刷）。
