## Context

全新空项目，目标产品是**网页版俄罗斯方块 4 人对战**。本 change 是里程碑 1——单机可玩地基。

约束：
- 俄罗斯方块对**输入延迟、旋转手感、锁定时机**极度敏感，体验完全取决于几个机制细节（旋转 kick、锁定延迟、DAS/ARR）是否到位。
- 最终产品要联机，届时服务器需能验证客户端的消行/能量、并用相同种子重放出一致的棋盘。因此**所有游戏规则必须与 I/O（DOM、网络、计时器副作用）解耦**，抽到可被 client 与 server 共同 import 的 `shared/`。
- 无历史代码、无迁移负担。

利益相关方：玩家（手感与公平）、未来联机 change（依赖 shared 边界稳定）。

## Goals / Non-Goals

**Goals:**
- 交付一个**完整可玩**的单人俄罗斯方块（含现代手感所需全部机制）。
- 把游戏规则沉淀为**纯函数 / 无副作用**的 `shared/` 层，单测可覆盖，且 server 可直接复用。
- 让循环具备**确定性**：相同种子 + 相同输入序列 → 相同棋盘，为联机重放铺路。
- 速度曲线、能量积累按需求参数化，可后续平衡。

**Non-Goals:**
- 联机、房间、对战、攻击（见 proposal 非目标）。
- 计分、排行榜、账号、持久化。
- 移动端 / 触屏（仅键盘）。

## Decisions

### D1. shared/ 层：纯逻辑、零 I/O 依赖
游戏规则（方块、旋转、棋盘、消行、7-bag、速度曲线）全部放在 `shared/`，**不 import 任何 DOM / 网络 / `requestAnimationFrame` / `Date.now()`**。时间以"传入的 elapsed ms"为参数，随机以"传入的 RNG"为参数。
- **为什么**：联机时 server 要用同一份代码做权威验证与确定性重放；纯函数才可单测、可重放。
- **替代方案**：规则直接写进 client 组件 → 联机时必须重写一份、双份维护、易不一致。否决。

### D2. 旋转系统：SRS + Wall Kick（标准 Tetris Guideline）
采用 Super Rotation System，含 I/O 方块的特殊旋转中心和标准 wall kick 偏移表（JLSTZ 一套、I 一套、O 不 kick）。
- **为什么**：事实标准，玩家手感预期即此；wall kick 是 T-spin、贴墙旋转等高级操作的前提，平衡对战时会用到。
- **替代方案**：简单旋转（无 kick）→ 贴墙无法旋转、手感差。否决。直接照搬公开的 SRS kick 表，不自创。

### D3. 随机：可种子化 7-bag
洗一袋 7 种 → 依次出 → 空袋再洗。RNG 接口可注入种子（如 mulberry32）。
- **为什么**：体验稳定（不会连续同种）；**可种子化**是联机"共享种子同步所有人方块序列"的基石。
- **替代方案**：纯随机 Math.random() → 不可复现、无法联机同步。否决。

### D4. 循环：固定时间步 + 逻辑/渲染解耦
渲染走 `requestAnimationFrame`（约 60fps，只画当前状态）；逻辑走**固定步长**（如 1000/60 ms 一 tick），由"累积实际时间"驱动下落。两者解耦。
- **为什么**：固定步长 = 确定性 = 可重放；渲染抖动不影响逻辑一致性。
- **替代方案**：每帧用 delta 直接推进逻辑 → 非确定性，联机重放时浮点漂移导致分歧。否决。

### D5. 锁定延迟 + 重置上限（Lock Delay with move count cap）
方块触底后进入 Locking，给 ~500ms 微调窗口；玩家移动/旋转重置延迟，但**最多重置 N 次**（如 15），超限或超时即锁定。
- **为什么**：无锁定延迟则贴底操作极难受；无重置上限则可无限拖延刷分。
- **替代方案**：触底立即锁定 → 手感差。否决。

### D6. 输入手感：DAS / ARR
按住 `A`/`D`：先等 DAS（~150ms）再以 ARR（~30ms/格）连续移动；`S` 软降按住同理连续下落。DAS/ARR/软降速率均设为常量、可调。
- **为什么**：没有 DAS/ARR 的左右移动笨重不可用，这是俄罗斯方块手感的硬需求。
- **替代方案**：仅在 keydown 触发一次 → 无法快速横移。否决。

### D7. 速度曲线：纯函数 + 参数化
`speed(elapsedMs, config) → rowsPerSecond`，纯函数。默认配置：`baseSpeed` 为 1.0× 含义（如基础 1 行/秒），前 3 分钟保持 1.0×，之后每过 1 分钟 +0.1×（即 3~4min=1.1×，4~5min=1.2×…）。
- **为什么**：纯函数可在 client/server 共用且确定；参数化便于平衡（用户曲线偏温和，后续可调）。
- **替代方案**：阶梯式查表 → 灵活性差。当前线性增长更直观，保留为配置。

### D8. 能量：积累与释放解耦
本 change 只实现**积累**（消行得分、上限 10、纯函数 `gainEnergy(linesCleared) → delta`）与 HUD 显示；**释放/消耗**（按 U 攻击）留待阶段二。能量状态放在 session，规则放 `shared`。
- **为什么**：积累逻辑可单机验证、不依赖联机；提前定义边界，阶段二只新增"释放"需求（将成为本 capability 的 MODIFIED Requirement）。

### D9. 渲染：Canvas 2D
单机棋盘用 Canvas 2D 绘制（棋盘、方块、next、hold、ghost、能量条）。
- **为什么**：10×20 网格渲染 Canvas 2D 完全够用，无需 WebGL 复杂度；后续联机也只是多画几个小预览棋盘。
- **替代方案**：DOM 网格 / WebGL → DOM 重绘开销大、WebGL 过度工程。否决。

### D10. 能力（Capability）边界
划分为四个独立可演进的 spec：`tetris-core`（纯规则）、`speed-progression`（曲线）、`energy-accumulation`（积累）、`single-player-session`（整合运行）。`single-player-session` 组合前三者 + 循环/输入/渲染。
- **为什么**：每个边界独立可测、可被未来 change 单独扩展（尤其 energy 会加"释放"、core 会被 server 复用）。

## Risks / Trade-offs

- **[锁定延迟被无限利用刷分]** → 设移动/旋转重置次数上限（D5）。
- **[DAS/ARR/锁定延迟默认值不当导致手感差]** → 全部集中为命名常量、可一键调参；首版用社区公认值（DAS≈150ms、ARR≈30ms、锁定≈500ms、重置上限≈15）。
- **[SRS wall kick 表抄错]** → 直接采用公开标准 SRS 表，配单测覆盖每种方块每个旋转方向的 kick 场景。
- **[固定时间步 + 解耦增加实现复杂度]** → 值得，确定性是联机的地基；用简洁的 accumulator 模式实现。
- **[速度曲线温和，高水平玩家嫌慢]** → 参数化，房间/难度可选更高曲线；本 change 先按需求默认值。
- **[纯函数要求禁用 `Date.now()`/`Math.random()`]** → 计时由调用方传入 elapsed，随机由注入的种子化 RNG 提供；这是约束也是优势。

## Open Questions

- **Hold 冷却**：交换后是否允许立刻再次 Hold 同一回合？（建议：不允许，与下一块同样需锁定后才能再 Hold——标准做法。首版按此实现。）
- **T-spin 识别**：本 change 是否识别 T-spin？（建议：暂不识别、不给额外奖励，但 core 的旋转/kick 已支持其存在；留作阶段二平衡项。）
- **基础下落速度具体值**：1.0× 定义为多少行/秒？（建议 1.0 行/秒起步，即首格约 1s；可调。）
- **暂停**：是否实现暂停？（建议实现，P 键或失焦自动暂停，单机常用。）

## Migration Plan

全新项目，无历史代码与线上流量。直接按 `shared/` → `client/game/` → `client/ui/` 顺序落地即可，无回滚需求。联机 change 将在本 change 产出的 `shared/` 边界上扩展，不破坏既有契约。
